Backend Integration Guide

How to connect a real database and email server to Daily Activity Monitor.

Why a Backend Is Needed

This website currently stores all data in localStorage — data that lives only in the user's browser. A backend server is needed to:

  • Store user accounts permanently in a real database
  • Send real verification and invitation emails
  • Share group data across different devices and users
  • Keep data safe if a user clears their browser
  • Allow any member to log in from any device and see their groups

Option 1 — MS Access ⚠ Not Recommended for Web

MS Access is a desktop database, not a web database. It cannot connect directly to a website hosted on the internet. To use it, you would need:

  • A Windows server running IIS (Internet Information Services)
  • An ASP.NET or Classic ASP middleware layer to bridge Access to the web
  • An ODBC driver configured on the server
  • A paid Windows hosting plan

Why it's not recommended

MS Access is not designed for concurrent web traffic, has poor security for public-facing websites, and is difficult to scale. The options below are free, easier, more secure, and widely used by modern web apps.

Option 2 — Firebase (Google) Free ★ Recommended Easiest

Firebase is a free cloud platform by Google. It provides a real database, user authentication with built-in email verification, and works directly with this HTML/JS site — no separate server required.

What Firebase provides

  • Firebase Authentication — handles registration, login, email verification, and password reset automatically
  • Firestore Database — stores users, groups, and activities in real-time
  • Free tier — up to 10,000 authentication events/month and 1 GB database storage at no cost

How to set it up (step by step)

  1. Go to firebase.google.com and sign in with a Google account
  2. Click "Add project" → name it "daily-activity-monitor" → click Continue
  3. In the project, click "Authentication" → "Get started" → enable Email/Password
  4. Click "Firestore Database" → "Create database" → choose your region → start in test mode
  5. Click the </> (Web) icon to register your app → copy the firebaseConfig object
  6. Add this to the <head> of every HTML page:
    <script src="https://www.gstatic.com/firebasejs/10.x.x/firebase-app.js"></script>
    <script src="https://www.gstatic.com/firebasejs/10.x.x/firebase-auth.js"></script>
    <script src="https://www.gstatic.com/firebasejs/10.x.x/firebase-firestore.js"></script>
  7. Replace localStorage calls in register.js, login.js, and session.js with Firebase SDK calls
  8. Firebase Auth sends real verification emails automatically — remove the simulation panel from verify-pending.html

What changes in the code

  • register.js → call createUserWithEmailAndPassword() then sendEmailVerification()
  • login.js → call signInWithEmailAndPassword() and check user.emailVerified
  • session.js → use onAuthStateChanged() instead of localStorage
  • create-group.js → write group to Firestore instead of localStorage
  • group-dashboard.js → query Firestore for groups

Option 3 — Supabase Free Easy

Supabase is an open-source Firebase alternative. It uses a real PostgreSQL database and provides built-in Auth with email verification.

How to set it up

  1. Go to supabase.com and create a free account
  2. Create a new project and note your Project URL and anon key
  3. Go to Authentication → Providers → Email and ensure it is enabled
  4. Create tables: users, groups, group_members
  5. Replace localStorage calls with supabase.from('users').insert() etc.
  6. Replace email verification simulation with supabase.auth.signUp({ email, password }) — it sends a real email automatically

Option 4 — Node.js + PostgreSQL Free Tier

A custom backend gives you full control. You write a small server that handles registration, login, email, and data storage.

Tech stack

  • Node.js + Express — web server (handles API calls from the browser)
  • PostgreSQL — database (stores users, groups, activities)
  • Nodemailer + Gmail / SendGrid — sends real verification and invite emails
  • bcrypt — securely hashes passwords (never store plain text in production)
  • Railway or Render — free hosting for the Node.js server and database

How it connects

Your HTML/JS files call your Node.js server using fetch(). For example, registration calls POST /api/register and the server saves the user to PostgreSQL and sends a real email via Nodemailer. Login calls POST /api/login and the server returns a session token.

Users table (equivalent to Users.txt)

In PostgreSQL, the users table would look like this:

  • id — auto-generated unique ID
  • first_name, last_name
  • email — unique
  • password_hash — bcrypt hash (never the plain password)
  • date_of_birth
  • phone
  • address_apt, address_street_number, address_street_name
  • province, country, postal_code
  • is_verified — boolean, set to true after email click
  • verify_token — UUID sent in verification email
  • created_at — timestamp

Summary — Which Should You Choose?

  • I want the simplest free solution that works today
    → Use Firebase. No server needed, works with your existing HTML files.
  • I want open-source and SQL-based
    → Use Supabase. Free PostgreSQL database, similar ease to Firebase.
  • I want full control over the server
    → Use Node.js + PostgreSQL on Railway. More work, full flexibility.
  • I want MS Access specifically
    → Only feasible with a Windows Server running ASP.NET — expensive and complex. Consider migrating to one of the above instead.
